# 機構範例藍圖（example-driven 實作 × 自動驗收）

> **這份給誰**：接手繼續做的 AI agent / 開發者（含 token 用完換手的情況）。
> **這份幹嘛**：把「要做哪些經典機構範例、每個怎麼算通過、怎麼自動驗、怎麼接手」一次講清楚。
> 搭配看：`docs/專案導覽.md`（全貌，§2 設計哲學、§7 進度藍圖）、`docs/HANDOFF-BLOCKS-ARCH.md`（齒輪技術細節）。

## 0. 核心工作方式：範例＝驗收標準

不要「每改一行就停下來請使用者開瀏覽器檢查」（太費時）。改成 **example-driven**：

> **挑一個經典範例 → 實作 → 用 Node 腳本自動斷言它的「機構性質」→ 綠了才請使用者看一眼視覺/手感。**

把驗證從「每次改動」收斂成「每個範例一次目視」。原因：機構「對不對」的大部分（解得穩、自由度、轉速比、
行程、平行、軌跡）都走**純核心**（`js/core` + `js/multilink`），**我能用 Node headless 自動驗，不需要人**。

### 哪些自動、哪些要人

| | 能不能自動 | 怎麼做 |
|---|---|---|
| 解得穩、無 NaN、無跳變 | ✅ 自動 | 掃 theta 0–360，每幀檢查 |
| 自由度 F=1 | ✅ 自動 | Gruebler `F=3(n−1)−2j₁−j₂`，或驗軌跡單參數 |
| 機構性質（轉速比/行程/平行/倍率/直線/閉合） | ✅ 自動 | 每範例一條數值斷言 |
| 視覺（畫面/z序/齒形/紅環/顏色） | ❌ 要人 | 載入範例看一眼 |
| 互動手感（拖曳/吸附/點擊） | ❌ 要人 | 操作一次 |

## 1. 自動驗證怎麼跑（已驗證可行，零相依）

- 測試放 `test/`，檔名用 **`.mjs`**。跑法：**`node test/<範例>.mjs`**。
- Node ≥ 22 會**自動偵測 ESM 語法**，所以 `.mjs` 直接 `import` 真正的核心 `.js` 就能跑——
  **不需要旗標、不需要 package.json、不裝任何套件**（完全合「零相依／無 build」）。
- `test/_harness.mjs`：共用工具（`sweep` 編譯+掃 theta 求解、`angle`/`unwrapSum`、`check`/`report`）。
  走的是 `compileTopology`（`js/core/topology.js`）+ `solveTopology`（`js/multilink/solver.js`，blocks 那條 bodyJoint 路徑）。
- **範本**：`test/gear-pair.mjs`（已通過）。新範例**照它複製改寫**即可。

```bash
node test/gear-pair.mjs
# PASS ✅ 全程解得穩 / PASS ✅ 轉速比 = −N驅動/N從動 且反向 (-0.7500)
```

## 2. 範例清單與狀態

按「功能」排（小孩理解機構是「這個讓那個動」）。**現在能做**＝用現有零件即可（連桿/銷/地錨/滑塊/齒輪/馬達）。

| # | 範例 | 教的概念 | 用到零件 | 現在能做？ | 狀態 |
|---|------|----------|----------|-----------|------|
| 1 | 曲柄搖桿（四連桿） | 轉→擺動；Grashof | 連桿+銷+地錨+馬達 | ✅ | ✅ 自動驗收過（Grashof + 曲柄整圈 + 搖桿擺動） |
| 2 | 平行四邊形連桿 | 保持姿態/平行 | 同上 | ✅ | ✅ 自動驗收過（穩定工作區間內輸出/輸入方向差保持常數） |
| 3 | 曲柄滑塊 | 轉↔直線；活塞 | +滑塊 | ✅ | ✅ 自動驗收過（滑塊沿軌道往復 + 行程 = 2×曲柄半徑） |
| 4 | 縮放儀 pantograph | 放大/臨摹 | 連桿+銷 | ✅ | ✅ 自動驗收過（手抓輸出點 + 雙軌跡 + k=2 倍率） |
| 5 | 急回機構（偏置曲柄滑塊） | 同距離行程用不同曲柄角度 → 速度不同 | 滑塊+連桿 | ✅ | ✅ 自動驗收過（前進/回程角度明顯不相等） |
| 6 | 近似直線連桿（Chebyshev） | 直線軌跡 | 連桿+銷 | ✅ | ✅ 自動驗收過（工作區間內耦合點 P 近似直線） |
| 7 | 嚙合齒輪對 | 變速/變向 | 齒輪+馬達 | ✅ | ✅ 自動驗收過（math）；視覺待人工一眼 |
| 8 | 減速齒輪列 | 大減速比 | 多齒輪 | ✅ | ✅ 自動驗收過（連乘齒比 + 相鄰齒輪反向） |
| 9 | 步行腿（Jansen） | 連桿走路 | 連桿+銷 | ✅（已有專頁） | ✅ 自動驗收過（足端軌跡閉合、連續、行程明顯） |
| 10 | 齒條齒輪 | 轉→直線 | +齒條 | ✅ | ✅ 自動驗收過（位移 = R·θ 純滾動、轉一圈行程 = 2πR）；齒形/相切待人工一眼 |
| 11 | 凸輪從動件 | 轉→上下 | +凸輪 | ⏳ 需凸輪 | ⬜（P3） |
| 12 | 皮帶輪傳動 | 傳遞旋轉/比例 | +皮帶輪 | ⏳ 需皮帶輪 | ⬜（P3） |

> 1–9 現在就能做；10 已落地齒條（rack 零件）。剩 11–12 跟著各自新零件（凸輪、皮帶輪）一起來。

## 3. 每個範例的「驗收條件」（自動斷言要寫什麼）

> 每個都先隱含通過「全程 `allValid`」。下面是該範例的**性質斷言**（= 用範例當驗收的關鍵）。

1. **曲柄搖桿**：輸入曲柄總轉角 ≈ ±360°（能整圈）；輸出搖桿總轉角 ≈ 0 但擺幅 > 0（來回擺、不整圈）；
   與 Grashof 預測（`S+L ≤ P+Q` → 最短桿可整圈）一致。
2. **平行四邊形**：輸出桿方向角 − 輸入桿方向角 = 常數（全程變異 < 1e-3 rad）。
3. **曲柄滑塊**：滑塊沿軌道投影位移在 [min,max] 往復；行程值對。
4. **pantograph**：手抓右端輸出點操作；兩支三點桿維持共線，輸出點 R = B + k·(追蹤點 P − B)，k 全程為常數（目前 k=2），P/R 可同時留下不同色軌跡。
5. **急回**：偏置曲柄滑塊的前進 / 回程 theta 區間比明顯 ≠ 1（量滑塊到兩端點的 theta 分割）。圖上看的是「同一段水平位移，一段占較多曲柄角度、另一段占較少曲柄角度」；馬達等角速度時，占角度少的那段就是快回。
6. **直線連桿**：✅ 已完成 → `test/chebyshev-linkage.mjs`；耦合點 P 在工作區間內 y 變化 < 容差（近似直線），且 x 行程足夠明顯。
7. **齒輪對**：✅ 已完成 → `test/gear-pair.mjs`（轉速比 = −N驅動/N從動、反向）。
8. **減速齒輪列**：✅ 已完成 → `test/reduction-gear-train.mjs`；末輪/首輪總轉角 = 連乘齒比，相鄰齒輪反向。
9. **Jansen**：✅ 已完成 → `test/jansen.mjs`；足端軌跡閉合且連續，且行程明顯（細部步態曲線留人工目視）。

10. **齒條齒輪**：✅ 已完成 → `test/rack-pinion.mjs`；齒條沿軸向位移 = R·θ（純滾動、線性），轉一圈行程 = 2πR，且只沿軸向平移。

（11–12 的斷言在對應零件落地時定：凸輪 `從動位移 = 輪廓函數`；皮帶輪 `ω比 = 輪徑反比、同向`。）

## 4. 接手步驟（cold start，逐範例）

1. 從 §2 表挑一個 `⬜` 的範例（建議照 1→2→3… 順序）。
2. **準備 fixture**：寫出該機構的 comps（plain JSON）。若 `js/blocks/examples.js` 的 `BLOCK_EXAMPLES`
   已有就重用；沒有就**順手也加進 BLOCK_EXAMPLES**（app 裡才載得到，給使用者目視）。
3. **寫測試** `test/<範例>.mjs`：複製 `test/gear-pair.mjs`，換 fixture 與 §3 的性質斷言。
4. `node test/<範例>.mjs` → **綠了才往下**；紅了自己修（多半是 fixture/參數，不要動核心數學）。
5. 請使用者做**一次**目視：載入範例、按播放、「長得對 / 手感對」嗎。
6. 通過 → 把 §2 狀態改 ✅、commit（含 test + 必要的 example/程式）。

## 5. 紅線 / 專案精神（務必遵守）

- **不改求解器數學**：`js/multilink/solver.js`、`js/core/topology.js` 既有 step 數學一行不動；只加新分派。
- **元件維持 plain JSON**（undo/存檔/分享靠序列化）；不可改成 class 實例。
- **零相依、純前端、無 build、無 npm**；test 用 `.mjs` + `node`，不裝套件。
- 每個新零件都先問「**這零件會誘發什麼錯誤？怎麼設計掉？**」（線上樂高哲學，見專案導覽 §2）。

## 7. 隱性機架 / 多孔機架規劃

現況：所有 `type:'fixed'` 的接點在求解器裡都屬於同一個 ground body。這已經是「隱性機架」，只是過去畫面沒有把它完整表達出來；例如急回機構的曲柄軸 `O` 和滑軌固定孔 `M1/M2` 其實都鎖在同一塊底座上。

### 7.1 第一刀：自動視覺化隱性機架（不改格式）

- 不新增存檔欄位、不改 solver。
- 直接掃描所有 fixed 接點，排除滑軌內部 rail 端點，但保留滑軌 mount 孔。
- 1 個固定孔：維持地面基線。
- 2 個或近似共線固定孔：畫成固定多孔桿，沿桿身補 8mm pitch 孔位。
- 3 個以上且不共線：畫成淡灰色機架板 / 多孔板，板內補 8mm pitch 孔陣列。
- 機架把手維持原本語意：拖曳把手會平移整組 fixed 接點。
- 這一刀只把「線上樂高」的視覺規範做出來；若舊檔固定孔不在 8mm 格點上，先不強制修正，避免破壞作品。
- 介面提供兩個全域選項：
  - `孔`：顯示 / 隱藏 LEGO 機架孔陣列；機架板或固定桿本體仍保留。
  - `鎖`：拖曳 fixed 孔或整個機架時吸附到 8mm pitch；一般自由接點與機構手抓點不受影響。

### 7.2 第二刀：正式 frame 零件（待第一刀目視驗收後）

預期新增元件型別：

```json
{ "type": "frame", "id": "Frame1", "holes": ["O", "M1", "M2"] }
```

初期語意：

- `frame` 只描述「哪些 fixed 孔屬於同一塊可見底座」，不改求解。
- 載入舊檔時若沒有 frame，仍用第一刀的隱性機架自動推導。
- 使用者未來可選取 frame，調整顏色、顯示樣式、或把固定孔分成不同機架群。
- 正式 frame 應遵守 LEGO Technic 規範：孔距 `pitch = 8mm`，孔座標以整數 `(u, v)` 表示，再由 `origin + rotate(angle) * (u*8, v*8)` 轉成世界座標。

### 7.3 後續範例

第一刀驗收後，可新增或調整範例來凸顯機架：

- **急回機構**：O/M1/M2 顯示成三孔機架板。
- **曲柄滑塊**：曲柄軸與滑軌座顯示成同一個固定底座。
- **機架反置範例**：同一組連桿，換不同孔當 frame，觀察運動型態改變。
