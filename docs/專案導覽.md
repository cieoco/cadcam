# 專案導覽（入門用，給人看的）

> 這份的目的：用最白話的方式，讓你在 5 分鐘內重新抓到「這專案是什麼、怎麼蓋的、做到哪、能往哪走」。
> 細節技術文件在最後一節有地圖；這份只負責「看懂全貌」。

---

## 1. 這是什麼？（用途）

一個**在瀏覽器裡設計「連桿機構」的工具**。連桿機構就是用桿子、轉軸、滑軌兜出來、會動的機械結構
（像汽車雨刷、伸縮燈、夾爪那種）。

你可以用它做三件事：

1. **畫** —— 擺零件、接起來，組成一個機構。
2. **模擬** —— 按播放，看它真的動起來、會不會卡住。
3. **輸出** —— 匯出 DXF（拿去雷射切割/CNC）或 `mechanism.json`（交給姊妹工具 `arm` 做 3D 組裝）。

它有**兩種使用面**：

- **機構工作頁**：選一種現成機構（四連桿、曲柄滑塊、齒條齒輪、平行四邊形、Jansen 腿…），調參數看它動。
- **blocks 積木沙盒**（`blocks.html`）：像樂高一樣**自由擺零件**自己兜機構。這是目前主力發展的方向，
  齒輪也是先在這裡長出來的。

**設計定位**：教學 / maker 用，UI 是繁體中文，強調「不用安裝、打開瀏覽器就能玩」。

---

## 2. 怎麼跑起來？

沒有安裝、沒有 npm、沒有 build。只要用 HTTP 開：

```bash
python -m http.server 8000
# 然後瀏覽器開 http://localhost:8000/mechanism.html  （機構工作頁）
#            或 http://localhost:8000/blocks.html      （積木沙盒）
```

（一定要用 HTTP，不能直接點開檔案——因為用了瀏覽器的 ES6 module 載入，`file://` 不支援。）

線上版掛在 GitHub Pages，所以 push 到 `main` 就等於更新線上 demo。

## 3. 怎麼蓋的？（心智模型，記這幾條就夠）

### (a) 純前端、零相依

整個專案就是一堆 `.js` 直接在瀏覽器跑，沒有打包工具、沒有套件。改完存檔、重整網頁就生效。

### (b) 最重要的一條界線：算數學的 vs 畫畫面的

```
   js/core/  ──「算」──   純函式，做數學/求解，不碰畫面
      │
      │  中間一個「引擎門面」mechanism-engine.js 把兩邊接起來
      │
   js/ui/    ──「畫」──   只管 DOM、滑鼠點擊，不做數學
```

> 規則：**core 不准碰畫面，ui 不准放求解數學。** 改東西時守住這條線，bug 會少很多。

### (c) 機構是「登錄表」驅動的

`js/mechanism-config.js` 像一張總表，每一種機構是一筆設定，寫好「它有哪些參數、用哪個 solver/畫圖/零件模組」。
**加一種新機構 = 在表裡加一筆 + 寫對應的三個檔（solver / visualization / parts）。** 不用改核心。

### (d) blocks 積木沙盒怎麼運作

零件種類：**地錨、連桿、三點桿、滑軌、齒輪、馬達**。你把它們擺上去、接起來，
底層會即時編譯成一個「拓樸（topology）」，丟給 **multilink 的求解器**算出每個關節的位置。

那個求解器用的是「**constructive geometry**（逐步建構幾何）」：從已知的固定點出發，
一個一個推算出其他點該在哪——適合單一自由度的機構。**這套數學是紅線，盡量別動。**

### (e) 每個機件 = 「資料卡」＋「行為表」（不是傳統 OOP 物件）

概念上，每種機件（齒輪、連桿…）就是一包「**屬性 + 行為**」：屬性（齒數、位置…）、
行為（怎麼畫、怎麼求解）。但實作上我們**故意把資料和行為拆開放**，不綁成一個 class 實例：

- **資料 = 純 JSON「資料卡」**，不帶任何方法。例如齒輪就是
  `{ type:'gear', id, p1, p2, teeth, module, mesh }`——它只是「我是齒輪，這些數字」，本身不會畫、不會算。
- **行為 = 按 `type` 查的「對照表」**，跟資料分開：`PART_TYPES[type]`（有哪些點/參數）、
  `PART_DRAW[type]`（怎麼畫）、未來還有 compile/solve 的表（怎麼編譯/求解）。
  要對一張卡做事時，就用它的 `type` 去**查**對應行為。

**比喻**：像桌遊——棋子是樸素塑膠塊（資料卡：「齒輪棋子，齒數12」），規則寫在另一本按棋子種類分類的
規則書（行為表）。棋子身上不帶規則；要動它時去翻規則書。OOP 則相反：每顆棋子是「聰明棋子」，自己肚子裡裝著規則。

**為什麼故意拆開？（紅線）** 因為這工具一直要**復原 / 存檔 / 分享連結**，這三件都得把整個機構變成
純文字（JSON）存起來再讀回。純資料卡天生就能變 JSON、讀回完全一樣；聰明物件（class 實例）肚子裡的方法
沒辦法乾淨地序列化。所以有條紅線：**「元件維持 plain JSON，不可改成 class 實例」**。

> 這種「資料 + 按型別查行為」的風格叫**資料導向 / 實體＋系統**，遊戲引擎和函數式語言常這樣做，
> 跟傳統 OOP 是兩條路。而「加機件＝加表項」之所以成立，正是因為行為都集中在這些對照表裡。

### (f) 「算」是怎麼運作的（求解器）

求解器要回答的只有一個問題：**「固定點在哪、馬達轉了幾度——其他每個關節會跑到哪？」**
按一次播放就要算出所有關節新位置，才畫得出下一幀。

用的方法是 **constructive geometry（逐步建構幾何）**，像**圓規＋直尺作圖**、像「按順序連連看」：

1. **先標出一開始就知道的點**：地錨（位置固定）、馬達曲柄端點（馬達角度決定它甩到哪）。
2. **再一個一個推出其他點**，核心招式「**兩圓交點**」（機構學叫 dyad）：某關節用兩根固定長度的桿接到
   兩個已知點 A、B → 它離 A、離 B 各是定值 → 以 A、B 為圓心各畫一個圓，**兩圓相交處**就是這關節。
3. **重複**直到所有點都推出來——**不用解聯立方程式**。

**為什麼能這樣**：因為是**單一自由度**機構——固定機架 + 轉一個馬達角 → 其餘全被決定，只有唯一一組合理位置。
（這也是齒輪一定要「兩端都釘」的原因：沒釘死就不是單一自由度，推不出來。）

**「編譯」的角色**：你不能推一個點，除非它的參考點先被推出來。所以求解前要**排對順序**——
編譯就是把你擺的零件轉成一張**有順序的步驟表（topology / steps）**：

```
步驟1  ground      P1, P2          ← 已知（地錨）
步驟2  input_crank P3 = 馬達甩出的曲柄端
步驟3  dyad        P4 = 圓(P3,長a) ∩ 圓(P2,長b)
...
```

求解器照這張表由上往下一步步算。**「資料卡＋行為表」最後就是被編譯成這張步驟表餵進來。**
齒輪在這裡是一種 step：「角度被綁住的曲柄」——輸出銷 = 中心 + 半徑×方向(齒輪角)，齒輪角 = 馬達角×齒比（從動輪反向）；
算出來後下游連桿把它當已知點繼續推。

**兩個會卡住的狀況**（模擬時會遇到）：

- **分支(branch)**：兩圓其實有兩個交點（手肘上彎/下彎）。求解器靠「上一幀在哪」挑連續的那個，才不會突然翻面。
- **死點(dead point)**：兩圓剛好相切（只碰一點），機構走到「不知道下一步往哪彎」的姿勢 → 跳出死點橫幅提醒。

> 一句話：**「算」＝先把零件編譯成有順序的步驟表，再像圓規作圖一樣、用「兩圓交點」從已知點一路推出所有關節。**
> 整套數學住在 `js/core/`（純函式、不碰畫面）；blocks 沙盒走的是 multilink 的這個求解器（`js/multilink/solver.js`）。

## 4. 做到哪了？（進度）

### 已經穩的

- **多種現成機構**：四連桿、曲柄滑塊、齒條齒輪、平行四邊形、通用 multilink、Jansen 腿、2D bar drawer。
- **blocks 沙盒的大重構**：把原本一大包的程式切成乾淨的小模組（畫圖、狀態、面板、工具、輸入…），
  並把效能瓶頸（每幀重畫整棵畫面）修掉。這部分已收斂、已驗證。

### 最近在做的：把 blocks 變成「機構繪製＋模擬系統」，開始加「基礎機件」

核心想法：**未來要陸續加入很多基礎機件（齒輪、齒條、凸輪、彈簧…），每個機件「連求解器一起進來」，
而且加一個機件 ≈ 加一張表項，不用改核心。** 這就是「機件外掛」架構。

**第一個真機件＝齒輪，已經落地：**

- 齒輪以**嚙合對**為單位（一個驅動、一個從動，會反向、按齒比轉）。
- **安裝方式比照連桿**（內部代號「方案乙」）：放下時是**浮動**的，由你**把齒輪中心拖去地錨/接點**才固定，
  跟接一根桿子一樣的操作邏輯。
- **要馬達才會轉**：沒給動力的齒輪就是個靜止的輪子（跟桿件一致）；要轉就把馬達放到驅動輪中心。
- 拖動時會**自動維持咬合**；如果硬把兩軸釘在對不上咬合距離的地方，齒輪會顯示**紅色虛線環**提醒。

**順手做的架構鋪路**：開始把「畫零件」的分派改成一張登錄表 `PART_DRAW`
（齒輪、三點桿已進表）。意思是日後加新機件，畫法就是「在表裡加一筆」。

> 如果你之前覺得「搞不清楚在忙什麼」——大方向就是這句：**讓齒輪這種機件，能像加表項一樣加進來。**

## 5. 骨子裡的願景：把它當機構學的「驗證機」

> 表面上這是「畫機構、看它動」的工具；**骨子裡，目標是用一個簡單的 UI 去詮釋機構學的核心概念，
> 而且能重現、能驗證。實用導向，不碰太特例的東西。**

### 機構學其實在問三件實用的事

1. **它會不會動、動得對不對？**（能不能成為一個機構）
2. **它怎麼動？**（位置、軌跡、速度）
3. **它好不好用？**（會不會卡、力傳得順不順）

### 「詮釋 / 理解 / 驗證」剛好對到工具的三件事

| 機構學 | 這個工具 |
|--------|----------|
| **詮釋**（把概念畫成圖） | 擺零件、接關節 |
| **理解**（看它怎麼動） | 按播放、看軌跡 |
| **驗證**（理論預測 vs 實際） | **公式先算一個答案 → 模擬跑出來 → 兩邊對照** |

最後一格是核心：很多機構學概念可以**「先用公式預測、再用模擬確認」**——這台工具天生就是做這件事的。
這就是「可以重現驗證」的精髓，也是這個願景的殺手鐧：**預測 → 模擬 → 確認。**

### 實用核心概念（麵包奶油，附「怎麼驗證」＋工具現況）

| 概念 | 一句話 | 怎麼驗證 | 工具現況 |
|------|--------|----------|----------|
| **自由度 DOF** | 要幾個輸入才能完全決定動作；`F=1` 才是實用機構，`0`=結構、`≥2`=太鬆 | 工具知道桿數/關節數 → 即時算 `F` 顯示；也解釋「為什麼解不出來」 | 尚未顯示（**最該做**） |
| **Grashof 條件** | 四連桿 `S+L ≤ P+Q` → 有桿能整圈轉 | 量四長 → 預測「最短桿整圈轉」→ 播放看它真的轉一圈 | 尚未做 |
| **耦合曲線/軌跡** | 連桿上某點走過的路徑 | 設軌跡點看曲線，對照預期 | **已有**（設軌跡點） |
| **死點 + 傳動角** | 共線會卡；傳動角越近 90° 力越順 | 算整圈傳動角看最小值，對應「哪裡沒力/快卡」 | 死點**已有**；傳動角未做 |
| **傳動比** | 齒輪 `ω從動/ω驅動 = −N驅動/N從動` | 模擬量轉速比，對照齒數比 | **已有**（齒輪） |
| **機架反置** | 換一根當機架 → 變不同機構 | 把不同點設地錨，看動作怎麼變 | **已有**（設地錨＝選機架） |

> 急回特性、瞬心速度分析等更進階的先不碰（不挑特例）；上面六個就是麵包奶油。

### 這個願景怎麼指引開發

工具已經會「畫了會動」；願景是讓它再進一步——**畫了會動、還告訴你機構學為什麼這樣動、並當場驗證給你看。**
最能彰顯這點、又最現成的是 **自由度即時顯示** 與 **Grashof 檢查**（公式乾淨、是/否明確、模擬能當場確認，
而且工具已握有所需資料）。

## 6. 未來可以往哪走？（走向）

由近到遠：

1. **把機構學概念做進 UI**（呼應 §5 願景）：**自由度即時顯示**（順便解釋解不出來的原因）、
   **Grashof 檢查**、傳動角——「公式預測 → 模擬確認」最漂亮的幾個。
2. **把更多機件加進來**：齒條（架構已留位）、凸輪、彈簧…。每加一個 ≈ 一張表項 + 它的畫法/求解。
3. **把「求解」也做成登錄表**：目前「畫」已經表化了；「編譯成拓樸」和「求解」還是寫死的型別判斷。
   這兩個碰到核心檔案，要小心、一刀一型別慢慢做、每刀都要在瀏覽器驗證。
4. **教學 / 分享導向**：簡化課堂操作、分享連結讓學生互相打開、範例與學習卡。
5. **長線願景**：用自然語言描述 → AI 生成可用的機構（LLM 出設計、求解器當裁判，反覆迭代）。
